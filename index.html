Current Pain Points

Three instability zones to fix

After tracing the codebase, three distinct problems stand out as the root causes behind the failing tests, broken dev mode, and deployment friction.

Critical bug #1
Single-thread constraint vs Tokio test harness
metacall-rs requires initialize() and destroy() on the same OS thread. Tokio's multi-threaded test runner violates this — causing most integration tests to deadlock or panic before the test logic even runs.
Dev experience #2
Dev mode errors don't surface to overlay
The watch/hot-reload path swallows errors from the metacall-rs layer. Developers see a blank page or silent failure instead of a useful error overlay, making the dev loop painful to iterate on.
Error handling #3
Error crate gaps across crate boundaries
The thiserror-based error types don't propagate cleanly between the pages, html-generator, and web-server crates. Errors deep in the FFI bridge often surface as opaque panics instead of typed, actionable messages.
The root cause — thread isolation problem Rust
// metacall-rs REQUIRES init + destroy on the SAME thread // Tokio spawns tests on arbitrary worker threads → deadlock // ❌ current test setup (fails) #[tokio::test] async fn test_render_page() { metacall_initialize(); // thread A let html = render_page("/home", json!({})).await; metacall_destroy(); // may run on thread B → panic } // ✅ proposed fix — pin to dedicated thread #[test] fn test_render_page() { let rt = Runtime::new_current_thread(); rt.block_on(async { metacall_initialize(); let html = render_page("/home", json!({})).await; metacall_destroy(); // guaranteed same thread assert(html.is_ok()); }); }

GSoC 2025 Proposal

12-week stabilization plan

Roughly three phases: fix the test harness and failing tests first, then stabilize dev mode, then polish error handling and CI/CD for production readiness.

Pre-GSoC · Now
Codebase study + architecture mapping
Traced all crates, identified the FFI thread constraint, built this demo, sent mentor intro.
Week 1–2 · Community bonding
Reproduce all failing tests + document root causes
Set up local MetaCall dev env (Nix flake), run test suite, log each failure against crate + thread context.
Week 3–4
Fix metacall-rs single-thread test harness
Introduce a dedicated thread wrapper for MetaCall lifecycle in tests, ensure init/destroy on same OS thread.
Week 5–6
Stabilize integration + unit test coverage
Fix remaining failing tests, add missing coverage for pages routing, html-generator output, and cache TTL logic.
Week 7–8
Dev mode: hot reload + error overlay
Wire error propagation from metacall-rs through to the dev overlay. Ensure watch restarts cleanly on TSX change.
Week 9–10
Error crate: typed errors across all boundaries
Audit ? propagation from ts_loader → metacall-rs → pages → web-server. Replace opaque panics with thiserror variants.
Week 11–12
CI/CD hardening + production build validation
GitHub Actions matrix for Linux/macOS, Docker build smoke test, end-to-end deploy validation. Final documentation pass.
